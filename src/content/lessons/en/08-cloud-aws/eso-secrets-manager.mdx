In module 5 you got the literal secret out of git with ESO using the **fake
provider**: you learned the mechanics (SecretStore = where, ExternalSecret = what)
without cloud credentials. This is the promise we left pending: **the same pattern,
but against a real backend, AWS Secrets Manager.** The elegant part is that the
`ExternalSecret` and the app's consumption **don't change** —only the SecretStore's
`provider` does.

## Step 1: create the secret in Secrets Manager

First the secret lives in AWS. You create it with the CLI:

```bash
aws secretsmanager create-secret \
  --name prod/sample-app/api-key \
  --secret-string "super-secret-prod-key"
```

## Step 2: authentication from a NON-EKS cluster

Here's the key decision for you. ESO needs AWS credentials to read the secret, and
**how** it obtains them depends on where your cluster runs:

- **IRSA** (IAM Roles for Service Accounts) — a k8s ServiceAccount assumes an IAM
  role via OIDC, and ESO receives **temporary credentials that rotate on their
  own**. It's the ideal option... but it's **EKS-only**: it depends on the OIDC
  provider that EKS exposes. On your on-premise k3s you **don't have it out of the
  box**.
- **IAM user with access keys in a k8s Secret** — you create an IAM user with a
  read-only policy on the secret, and store its access key + secret key in a
  Kubernetes Secret that the SecretStore references. **Pragmatic and works today on
  k3s**, but it's a static credential: **you must rotate it** with discipline.
- **OIDC federation / workload identity** — standing up your own OIDC provider so
  k3s federates identity with IAM without static keys. It's the "correct" path
  without EKS, but **advanced**; I only mention it so you know it exists.

For your k3s, the realistic route is the middle one: **IAM user + access keys in a
Secret, rotating**. First you create the Secret with the keys:

```bash
kubectl create secret generic awssm-secret \
  --namespace sample-prod \
  --from-literal=access-key=AKIA... \
  --from-literal=secret-access-key=...
```

## Step 3: the SecretStore (provider.aws)

The SecretStore declares the AWS provider, the service, the region, and where to
get the credentials from:

```yaml
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: aws-secrets
  namespace: sample-prod
spec:
  provider:
    aws:
      service: SecretsManager
      region: sa-east-1
      auth:
        secretRef:
          accessKeyIDSecretRef:
            name: awssm-secret
            key: access-key
          secretAccessKeySecretRef:
            name: awssm-secret
            key: secret-access-key
```

`accessKeyIDSecretRef` and `secretAccessKeySecretRef` point at the `awssm-secret`
Secret you created in step 2. On EKS with IRSA, the whole `auth` block disappears
—ESO uses the ServiceAccount's role.

## Step 4: the ExternalSecret

Identical to the fake-provider lab, except `remoteRef.key` now points at the
secret's name in Secrets Manager:

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: worker-secrets
  namespace: sample-prod
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets
    kind: SecretStore
  target:
    name: worker-secrets        # the k8s Secret ESO creates
    creationPolicy: Owner
  data:
    - secretKey: DB_PASSWORD        # the key inside the resulting Secret
      remoteRef:
        key: prod/worker/db-password   # the secret in Secrets Manager
```

ESO creates the `worker-secrets` Secret with the `DB_PASSWORD` key, and your
NestJS service consumes it via `envFrom.secretRef` just like always. Git only holds
these recipes; the value lives only in AWS and, materialized, inside the cluster.

## Cost and rotation

Secrets Manager costs **$0.40 per secret per month + $0.05 per 10,000 API calls**,
with a **30-day free trial** per secret. The `refreshInterval: 1h` controls how
many calls you make: a very aggressive interval multiplies the calls. Secrets
Manager supports **automatic rotation** (with a Lambda) for database credentials;
ESO re-syncs on its own per the `refreshInterval`, so rotating the value in AWS
propagates to the cluster without you touching anything.

## Recommendation for your scale: Parameter Store

Be honest about cost. For dozens of services, $0.40 per secret per month adds up. **AWS
Systems Manager Parameter Store** stores parameters (including KMS-encrypted
`SecureString`) and its **standard tier is free** (up to 10,000 parameters). ESO
supports it by changing one line: `service: ParameterStore` in the same
`provider.aws`. **For your scale, Parameter Store is the best default**; reserve
Secrets Manager for the few secrets that genuinely need managed automatic
rotation. Same ESO mechanics, a fraction of the cost.

from a real backend, at a cost that fits your budget.
