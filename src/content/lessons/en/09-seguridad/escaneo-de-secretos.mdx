SCA and SAST look at code and dependencies. But there is a category of leak that
is neither a bug nor a CVE: a credential someone **pasted into a file and
committed**. An `API_KEY`, a provider token, the `db-prod` password. It's the
most human mistake in the set, and the most expensive, because git doesn't
forget.

## Why secrets in git are forever

You already saw this in module 5 with External Secrets Operator: **git history
is forever**. Even if you delete the value in a new commit, the old commit is
still there, and `git log -p`, any clone, fork, or backup exposes it. The theme
runs through the whole course: the goal of the ESO lab was precisely to **get the
literal secret out of git** so git never sees it.

The consequence is harsh and worth memorizing: **a secret that touched git is
considered compromised**. It doesn't matter if the repo is private, if you
deleted it five minutes later, or if "nobody saw it." If it entered the history,
treat it as leaked.

## The tool: gitleaks

**gitleaks** is a secret scanner: it walks files and commits looking for patterns
that look like credentials (AWS keys, GitHub tokens, private keys, connection
strings) using regex and entropy rules. **trufflehog** is the best-known
alternative; it does the same and can additionally **verify** whether a
credential is still active against the provider. For this lab we use gitleaks
because of how direct it is.

gitleaks' key distinction is **what** it scans:

- **Working tree** — `gitleaks detect --no-git` checks the current files on disk
  as plain text, without looking at history. This is what a **pre-commit hook**
  runs: it stops you *before* the secret enters a commit.
- **Full history** — `gitleaks detect` (the default mode in gitleaks v8) walks
  **every commit** in the repo. This is what finds the secret that leaked three
  weeks ago and that `--no-git` would never see, because it's no longer in the
  current tree.

> Recent gitleaks versions also offer the `gitleaks git` (history) and `gitleaks
> dir` (working tree) subcommands, but `detect` is still valid and is what we use
> here.

## Finding it isn't fixing it: rotate, don't delete

Here is the mistake that costs dearly. You find the secret, open the file, delete
it, commit "remove leaked key," and breathe. **You fixed nothing.** The old
commit —the one with the credential— is still in the history. Anyone with access
to the repo (or to a clone or fork made earlier) runs `git log -p` and there it
is, intact.

The rule is unambiguous:

1. **Rotate the credential.** Go to the provider, **revoke** the leaked key and
   issue a new one. This is what actually closes the breach: the old credential
   stops working, so it no longer matters who has it.
2. **Optionally, rewrite history.** With `git filter-repo` (or BFG) you can purge
   the value from every commit, but this rewrites hashes and forces everyone to
   re-clone. It's hygiene, not the fix: even if you purge the history, earlier
   clones and backups **already have** the secret. That's why rotating is
   mandatory and rewriting is secondary.

Deleting from HEAD gives a false sense of safety. Rotating is the only thing that
truly invalidates what leaked.

## Where the scan runs: pre-commit and CI

The scan goes in **two places**, and they complement each other:

- **Pre-commit hook (local).** It runs `gitleaks` over the working tree *before*
  the commit is created. It's the cheap barrier: it catches the secret on your
  machine, before it touches history. It's preventive.
- **CI job (remote).** The local hook can be skipped (`--no-verify`) or not
  installed at all. CI scans the **full history** on every push and PR, so it's
  the safety net nobody can bypass. It's the real gate.

Pre-commit prevents; CI enforces. You need both: one is optional by design, the
other is mandatory.

## The CI job

The stable form today is the official gitleaks action. **Key**: it needs
`fetch-depth: 0` on checkout to clone the full history —otherwise it only sees
the latest commit and misses exactly what matters.

```yaml
  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          # Without this, the checkout is shallow and gitleaks can't see history
          fetch-depth: 0
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

The action fails the build when it finds a secret, so the PR stays blocked until
the team resolves it —by rotating the credential, not just deleting it from the
file.
