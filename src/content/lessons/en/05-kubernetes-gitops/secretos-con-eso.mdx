In the Kustomize lesson we flagged the anti-pattern: the overlay's `secretGenerator`
has `API_KEY=super-secret-dev-key` in plaintext, and that lives in git. This lab
fixes it with **External Secrets Operator (ESO)**: the cluster pulls the secret from
an external backend at runtime, and git never sees it.

## Why a secret in git is a breach-in-waiting

A secret committed to git is not "an oversight I'll fix later". **Git history is
forever**: even if you delete the value in a new commit, the old commit is still
there, and `git log -p` or any clone, fork, or backup exposes it. The industry rule
is blunt: **a secret that touched git is considered compromised and must be rotated.**
Deleting it is not enough. That is why the only real solution is for the secret to
**never enter** the repo.

## ESO's architecture

ESO splits the problem into two objects with clear responsibilities:

- **SecretStore** says **where** the secrets live: which backend (AWS Secrets
  Manager, Vault, Doppler...) and how to authenticate against it. It is the
  connection.
- **ExternalSecret** says **what** secret to fetch and what shape to materialize it
  in: from which backend key, into which Kubernetes Secret, with what field names.

ESO watches the `ExternalSecret`, goes to the backend via the `SecretStore`, and
**materializes a native Kubernetes Secret** with the values. Your app consumes that
Secret as always (via `envFrom.secretRef`); it does not know ESO exists. What lives
in git is the *recipe* (what to fetch from where), not the *value*.

## Step 1: Install ESO

ESO installs with Helm in its own namespace:

```bash
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  -n external-secrets --create-namespace
kubectl wait --for=condition=Ready pods --all -n external-secrets --timeout=180s
```

## Step 2: Create a SecretStore with the fake provider

For the lab we do not need a real backend: ESO ships a **fake** provider that serves
inline data, ideal for learning the mechanics without cloud credentials. The
SecretStore declares the provider and, in this case, the test data:

```yaml
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: fake-store
  namespace: sample-dev
spec:
  provider:
    fake:
      data:
        - key: "/sample-app/api-key"
          value: "super-secret-dev-key"
```

```bash
kubectl apply -f secretstore.yaml
```

In a real backend, the `value` would not be here: the SecretStore would only hold the
connection, and the secret would live in AWS/Vault. The fake provider puts it inline
just for the lab.

## Step 3: Create the ExternalSecret

The ExternalSecret references the SecretStore and describes the Secret to
materialize:

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: sample-app-secrets
  namespace: sample-dev
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: fake-store
    kind: SecretStore
  target:
    name: sample-app-secrets      # the k8s Secret ESO will create
    creationPolicy: Owner
  data:
    - secretKey: API_KEY          # the key inside the resulting Secret
      remoteRef:
        key: "/sample-app/api-key"  # where it fetches it from in the backend
```

```bash
kubectl apply -f externalsecret.yaml
kubectl get externalsecret -n sample-dev
```

When the ExternalSecret's status is `SecretSynced`, ESO will have created a Secret
`sample-app-secrets` with the key `API_KEY`. Verify it:

```bash
kubectl get secret sample-app-secrets -n sample-dev
```

## Step 4: Consume the Secret and remove the overlay literal

The app consumes the materialized Secret just like before, via `envFrom.secretRef` —
nothing changes in the Deployment, because the Secret's name (`sample-app-secrets`)
is the same:

```yaml
          envFrom:
            - secretRef: { name: sample-app-secrets }
```

What matters is the other side: **delete the `secretGenerator` from the overlay**.
That is the source of the secret in git, and ESO now replaces it:

```yaml
# k8s/overlays/dev/kustomization.yaml — DELETE this block:
# secretGenerator:
#   - name: sample-app-secrets
#     literals:
#       - API_KEY=super-secret-dev-key
```

Now git contains the SecretStore and the ExternalSecret (recipes, without values)
but **no plaintext secret**. The value exists only in the backend and, once
materialized, inside the cluster.

```bash
kubectl get externalsecret sample-app-secrets -n sample-dev -o wide
```

## Real backends: only the SecretStore changes

The elegant part of ESO is that the `ExternalSecret` and the app's consumption **do
not change** when you move to a real backend. The only thing that changes is the
SecretStore's `provider`:

- **AWS Secrets Manager** (we will see it in module 8): `provider.aws` with the
  region and an IAM role.
- **Doppler**: `provider.doppler` with a service token.
- **HashiCorp Vault**: `provider.vault` with the address and the auth path.

The ExternalSecret still says "fetch `/sample-app/api-key` into `API_KEY`": rotating
the secret becomes changing the value in the backend, and ESO re-syncs on its own per
the `refreshInterval`. Compared to encrypting secrets in git (SOPS, Sealed Secrets),
you have nothing encrypted to rotate or decrypt in the cluster: the secret was never
in git.
