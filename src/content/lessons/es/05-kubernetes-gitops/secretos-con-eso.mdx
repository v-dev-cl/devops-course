En la lección de Kustomize señalamos el anti-patrón: el `secretGenerator` del
overlay tiene `API_KEY=super-secret-dev-key` en texto plano, y eso vive en git.
Este lab lo arregla con **External Secrets Operator (ESO)**: el cluster trae el
secreto desde un backend externo en tiempo de ejecución, y git nunca lo ve.

## Por qué un secreto en git es una brecha en espera

Un secreto comiteado a git no es "un descuido que arreglo después". El **historial
de git es para siempre**: aunque borres el valor en un commit nuevo, el commit viejo
sigue ahí, y `git log -p` o cualquier clon, fork o backup lo expone. La regla de la
industria es tajante: **un secreto que tocó git se considera comprometido y hay que
rotarlo.** No basta con borrarlo. Por eso la única solución de verdad es que el
secreto **nunca entre** al repo.

## La arquitectura de ESO

ESO separa el problema en dos objetos con responsabilidades claras:

- **SecretStore** dice **dónde** viven los secretos: qué backend (AWS Secrets
  Manager, Vault, Doppler...) y cómo autenticarse contra él. Es la conexión.
- **ExternalSecret** dice **qué** secreto traer y con qué forma materializarlo: de
  qué clave del backend, hacia qué Secret de Kubernetes, con qué nombres de campo.

ESO observa el `ExternalSecret`, va al backend vía el `SecretStore`, y **materializa
un Secret nativo de Kubernetes** con los valores. Tu app consume ese Secret como
siempre (vía `envFrom.secretRef`); no sabe que ESO existe. Lo que está en git es la
*receta* (qué traer de dónde), no el *valor*.

## Paso 1: Instala ESO

ESO se instala con Helm en su propio namespace:

```bash
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  -n external-secrets --create-namespace
kubectl wait --for=condition=Ready pods --all -n external-secrets --timeout=180s
```

## Paso 2: Crea un SecretStore con el provider fake

Para el lab no necesitamos un backend real: ESO trae un provider **fake** que sirve
datos en línea, ideal para aprender la mecánica sin credenciales cloud. El
SecretStore declara el provider y, en este caso, los datos de prueba:

```yaml
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: fake-store
  namespace: sample-dev
spec:
  provider:
    fake:
      data:
        - key: "/sample-app/api-key"
          value: "super-secret-dev-key"
```

```bash
kubectl apply -f secretstore.yaml
```

En un backend real, el `value` no estaría acá: el SecretStore solo tendría la
conexión, y el secreto viviría en AWS/Vault. El provider fake lo pone inline solo
para el lab.

## Paso 3: Crea el ExternalSecret

El ExternalSecret referencia el SecretStore y describe el Secret a materializar:

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: sample-app-secrets
  namespace: sample-dev
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: fake-store
    kind: SecretStore
  target:
    name: sample-app-secrets      # el Secret de k8s que ESO va a crear
    creationPolicy: Owner
  data:
    - secretKey: API_KEY          # la clave dentro del Secret resultante
      remoteRef:
        key: "/sample-app/api-key"  # de dónde la saca en el backend
```

```bash
kubectl apply -f externalsecret.yaml
kubectl get externalsecret -n sample-dev
```

Cuando el estado del ExternalSecret sea `SecretSynced`, ESO habrá creado un Secret
`sample-app-secrets` con la clave `API_KEY`. Verifícalo:

```bash
kubectl get secret sample-app-secrets -n sample-dev
```

## Paso 4: Consume el Secret y quita el literal del overlay

La app consume el Secret materializado igual que antes, vía `envFrom.secretRef`
—no cambia nada en el Deployment, porque el nombre del Secret (`sample-app-secrets`)
es el mismo:

```yaml
          envFrom:
            - secretRef: { name: sample-app-secrets }
```

Lo importante es el otro lado: **borra el `secretGenerator` del overlay**. Esa es la
fuente del secreto en git, y ahora la reemplazó ESO:

```yaml
# k8s/overlays/dev/kustomization.yaml — ELIMINA este bloque:
# secretGenerator:
#   - name: sample-app-secrets
#     literals:
#       - API_KEY=super-secret-dev-key
```

Ahora git contiene el SecretStore y el ExternalSecret (recetas, sin valores) pero
**ningún secreto en texto plano**. El valor solo existe en el backend y, ya
materializado, dentro del cluster.

```bash
kubectl get externalsecret sample-app-secrets -n sample-dev -o wide
```

## Backends reales: solo cambia el SecretStore

Lo elegante de ESO es que el `ExternalSecret` y el consumo de la app **no cambian**
al pasar a un backend de verdad. Lo único que cambia es el `provider` del
SecretStore:

- **AWS Secrets Manager** (lo veremos en el módulo 8): `provider.aws` con la región
  y un rol IAM.
- **Doppler**: `provider.doppler` con un token de servicio.
- **HashiCorp Vault**: `provider.vault` con la dirección y el path de auth.

El ExternalSecret sigue diciendo "trae `/sample-app/api-key` hacia `API_KEY`": la
rotación del secreto pasa a ser cambiar el valor en el backend, y ESO re-sincroniza
solo según el `refreshInterval`. Comparado con cifrar secretos en git (SOPS, Sealed
Secrets), no tienes nada cifrado que rotar ni desencriptar en el cluster: el secreto
nunca estuvo en git.

<aside>

**Aplícalo de verdad.** Esto amerita enmendar el `PLAN.md` de tu repo de definiciones de Argo CD:
el plan original contemplaba SOPS para cifrar secretos en el repo; con ESO el
secreto sale del repo por completo. Cambia la sección de secretos del PLAN de
"cifrar con SOPS" a "ESO + SecretStore apuntando a AWS Secrets Manager", y deja los
ExternalSecret en los overlays como recetas versionadas.

</aside>
