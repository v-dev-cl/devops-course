En el módulo 5 sacaste el secreto literal de git con ESO usando el **provider
fake**: aprendiste la mecánica (SecretStore = dónde, ExternalSecret = qué) sin
credenciales cloud. Esta es la promesa que dejamos pendiente: **el mismo patrón,
pero contra un backend real, AWS Secrets Manager.** Lo elegante es que el
`ExternalSecret` y el consumo de la app **no cambian** —solo cambia el `provider`
del SecretStore.

## Paso 1: crea el secreto en Secrets Manager

Primero el secreto vive en AWS. Lo creas con la CLI:

```bash
aws secretsmanager create-secret \
  --name prod/sample-app/api-key \
  --secret-string "super-secret-prod-key"
```

## Paso 2: autenticación desde un cluster NO-EKS

Aquí está la decisión clave para ti. ESO necesita credenciales AWS para leer el
secreto, y **cómo** las obtiene depende de dónde corra tu cluster:

- **IRSA** (IAM Roles for Service Accounts) — un ServiceAccount de k8s asume un rol
  IAM vía OIDC, y ESO recibe **credenciales temporales que rotan solas**. Es la
  opción ideal... pero es **solo para EKS**: depende del proveedor OIDC que EKS
  expone. En tu k3s on-premise **no la tienes de fábrica**.
- **IAM user con access keys en un Secret de k8s** — creas un usuario IAM con una
  política de solo lectura sobre el secreto, y guardas su access key + secret key
  en un Secret de Kubernetes que el SecretStore referencia. **Pragmático y funciona
  hoy en k3s**, pero es una credencial estática: **debes rotarla** con disciplina.
- **OIDC federation / workload identity** — montar tu propio proveedor OIDC para
  que k3s federe identidad con IAM sin llaves estáticas. Es el camino "correcto"
  sin EKS, pero **avanzado**; solo lo menciono para que sepas que existe.

Para tu k3s, la ruta realista es la del medio: **IAM user + access keys en un
Secret, rotando**. Primero creas el Secret con las llaves:

```bash
kubectl create secret generic awssm-secret \
  --namespace sample-prod \
  --from-literal=access-key=AKIA... \
  --from-literal=secret-access-key=...
```

## Paso 3: el SecretStore (provider.aws)

El SecretStore declara el provider AWS, el servicio, la región y de dónde sacar
las credenciales:

```yaml
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: aws-secrets
  namespace: sample-prod
spec:
  provider:
    aws:
      service: SecretsManager
      region: sa-east-1
      auth:
        secretRef:
          accessKeyIDSecretRef:
            name: awssm-secret
            key: access-key
          secretAccessKeySecretRef:
            name: awssm-secret
            key: secret-access-key
```

`accessKeyIDSecretRef` y `secretAccessKeySecretRef` apuntan al Secret
`awssm-secret` que creaste en el paso 2. En EKS con IRSA, todo el bloque `auth`
desaparece —ESO usa el rol del ServiceAccount.

## Paso 4: el ExternalSecret

Idéntico al del lab fake, salvo que el `remoteRef.key` ahora apunta al nombre del
secreto en Secrets Manager:

```yaml
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: worker-secrets
  namespace: sample-prod
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets
    kind: SecretStore
  target:
    name: worker-secrets        # el Secret de k8s que ESO crea
    creationPolicy: Owner
  data:
    - secretKey: DB_PASSWORD        # la clave dentro del Secret resultante
      remoteRef:
        key: prod/worker/db-password   # el secreto en Secrets Manager
```

ESO crea el Secret `worker-secrets` con la clave `DB_PASSWORD`, y tu servicio
NestJS lo consume vía `envFrom.secretRef` igual que siempre. Git solo contiene
estas recetas; el valor solo vive en AWS y, materializado, dentro del cluster.

## Costo y rotación

Secrets Manager cuesta **$0.40 por secreto al mes + $0.05 por cada 10.000 llamadas
a la API**, con **30 días de prueba gratis** por secreto. El `refreshInterval: 1h`
controla cuántas llamadas haces: un intervalo muy agresivo multiplica las llamadas.
Secrets Manager soporta **rotación automática** (con una Lambda) para credenciales
de base de datos; ESO re-sincroniza solo según el `refreshInterval`, así que rotar
el valor en AWS se propaga al cluster sin que toques nada.

## Recomendación para tu escala: Parameter Store

Sé honesto con el costo. Para decenas de servicios, $0.40 por secreto al mes se suma. **AWS
Systems Manager Parameter Store** guarda parámetros (incluidos `SecureString`
cifrados con KMS) y su **tier estándar es gratis** (hasta 10.000 parámetros). ESO
lo soporta cambiando una línea: `service: ParameterStore` en el mismo
`provider.aws`. **Para tu escala, Parameter Store es la mejor opción por defecto**;
reserva Secrets Manager para los pocos secretos que de verdad necesiten rotación
automática gestionada. Misma mecánica de ESO, fracción del costo.

sincronizados desde un backend real, con un costo que cabe en tu presupuesto.
